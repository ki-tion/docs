#
前端性能优化

前端性能优化是个高频面试题，我以前负责项目时也做过一些性能优化，在这里总结一下。

性能优化通常分为首屏优化和运行时优化。首屏优化一般是在项目上线前进行，通常在打包时处理。 运行时优化是在代码运行时进行，这个比较宽泛且和业务相关。

# 首屏优化

大体思路是减小资源体积，减少请求次数，优化资源加载顺序，优化资源加载方式。

# 分析

主要是查看模块大小，分析哪些较大模块有优化空间。以webpack为例,webpack-bundle-analyzer插件可以分析打包后的模块大小。

// vue.config.js

const BundleAnalyzerPlugin = require('webpack-bundle-analyzer').BundleAnalyzerPlugin;
module.exports = {
    ...
    configureWebpack: (config) => {
        config.plugins.push(new BundleAnalyzerPlugin());
    }
    ...
}

# 不要生成map文件

map文件用于调试，生产环境不需要调试，关闭生成map文件。map文件体积较大，会延长打包时间，增加打包体积，且影响非常明显。 提供webpack示例，vite雷同

// vue.config.js
module.exports = {
    ...
    chainWebpack: config => {
        productionSourceMap:false,
    }
    ...
}

# 移除无效代码

没什么好说的，无用代码的直接删除就好。

# 公共模块

将公共模块提取出来，减少重复打包。且将之放到cdn上可以加快加载速度。 首先在webpack打包时不要打包公共模块。

// vue.config.js
module.exports = {
    ...
    externals: {
        'vue': 'Vue'
    }
    ...
}

然后在html中引入cdn资源。

<% if(NODE_ENV === 'production') { %>
<script src="https://cdn.bootcdn.net/ajax/libs/vue/3.0.0/vue.cjs.js"></script>
<% } %>

# 按需加载

引入一个包，但需要使用其中部分功能，此时全部加载显得浪费。只加载需要的部分，使用 Tree Shaking 技术。vue3 默认是开启的，这点不用多操心,在代码编写的时候注意import即可。

# 懒加载

懒加载是首屏优化中常用的手段。首次访问时本身就会加载核心的、全局的资源，这部分体积并不小，对于首屏用不到的资源尽量延迟加载。比如图片、视频、组件等。通常方案是路由懒加载、使用异步组件和动态导入等等方式，具体为编写代码时注意，这里按下不表。

# 代码分割&合并

对于某些包体积较大，一次全部加载将会导致加载时间过长；而对于某些包体积较小，单独加载又会增加请求次数。此时可以将这些包进行合并。

通常在打包工具中配置，以 webpack举例，vite思路相同。

// vue.config.js
...
configureWebpack: config => {
    if (process.env.NODE_ENV === 'production') { // 生产环境
        config.plugins.push(
            new webpack.optimize.LimitChunkCountPlugin({
                maxChunks: 10, // 来限制 chunk 的最大数量
                minChunkSize: 100000 //设置 chunk 的最小大小
                // maxChunkSize:10000
            })
        )
    }
}
...

# 压缩资源

压缩资源可以减小资源体积，加快加载速度。通常在打包工具中配置，以 webpack举例，vite思路相同。

// vue.config.js
...
chainWebpack: config => {
    // vuecli 3默认开启prefetch(预先加载模块)，提前获取用户未来可能会访问的内容， 在首屏会默认加载全部的js文件， 所以要关闭这个功能
    // 取消预加载和预取，顺便在这里直接写了
    // 这样在首屏时就不会加载多余的资源
    config.plugins.delete('preload') 
    config.plugins.delete('prefetch')
    if (process.env.NODE_ENV === 'production') {
        config.plugin('compressionPlugin').use(
            new CompressionPlugin({
                test: /\.(js|css|html)$/,// 匹配文件名
                threshold: 10240,
            })
        );
    }
}
···

这里需要注意，需要后端配合。我使用的nginx，配置如下：

...
http {
    ...
    gzip on; # 开启 gzip 压缩
    gzip_static on; # 若存在静态 gz 文件，则使用该文件
    gzip_min_length 10k; # 设置允许压缩的页面最小字节数
    gzip_buffers 16 8k; # 设置用于处理请求压缩的缓冲区数量和大小
    gzip_comp_level 4; # 设置压缩级别 1-9，数字越大，压缩后的大小越小，也越占用CPU，花费时间越长,可以根据需求选择
	# 对特定的 MIME 类型生效, 其中'text/html’被系统强制启用
    gzip_types application/javascript text/css font/ttf font/x-woff;
    gzip_vary on; # 是否在 http header中 添加 Vary:Accept-Encoding, on | off
    gzip_http_version 1.1; # 在 http/1.1 的协议下不开启压缩
    ...
}
...

# 静态资源

一是压缩静态资源，比如图片等；二是将静态资源放在cdn上，cdn会根据用户地理位置选择最近的服务器，从而加快资源加载速度。

对于一些体积很小的静态资源，比如图片，可以考虑使用base64编码，这样可以直接在html中引用，减少请求次数。

// vue.config.js

...
chainWebpack: (config) => {
    const imageRule = config.module.rule('images');
    imageRule
        .use('url-loader')
        .loader('url-loader')
        .tap((options) => {
            return {
                ...options,
                limit: 8192, // 小于 8KB 的图片将会被转换为 Base64
                fallback: {
                loader: 'file-loader',
                options: {
                    name: '[name].[hash:8].[ext]', // 输出文件的命名规则
                },
                },
            };
        });
}
...

# 资源加载顺序

这里主要利用script标签 defer async 属性。defer 和 async 的区别在于，defer 是在 html 解析完成之后执行，而 async 是在下载完成之后执行。因此，defer 的执行顺序和书写顺序一致，而 async 则是哪个先下载完就先执行哪个，因此执行顺序是不确定的。

# 优化DNS查询

尽量将资源放在同一个域名，使用DNS预解析。 在html中添加meta标签，告诉浏览器在加载页面时预解析域名。

<meta http-equiv="x-dns-prefetch-control" content="on">
<link rel="dns-prefetch" href="//www.example.com">

# 使用CDN。

CDN可以将资源缓存到全球的节点上，用户访问时可以直接从最近的节点获取资源。

# 效果

# 懒加载&动态加载

懒加载和动态加载是两种不同的概念，懒加载是页面加载时只加载当前需要显示的内容，而动态加载通常时根据业务逻辑动态加载内容。

懒加载的实现方式有很多种，比如使用IntersectionObserver API，或者使用vue的v-if指令。动态加载的实现方式也有很多种，比如使用import函数。

# keep-alive缓存

通常在没有性能问题时，尽量不要使用keep-alive。一是会增加内存，二是有时会给开发和调试带来麻烦

# 响应式数据

对于不需要响应式的数据，可以使用Object.freeze冻结对象，从而避免不必要的响应式处理。

# 防抖和节流

对于频繁触发的事件，可以使用防抖和节流来减少触发次数。至于具体实现方式，这里就不赘述了。

# 缓存

这是一个很大的概念，这里只简单说下。

  • 使用浏览器缓存。 分为强缓存和协商缓存，强缓存是浏览器直接从缓存中获取资源，而协商缓存是浏览器向服务器询问资源是否更新，如果更新则重新获取资源。
  • 接口缓存。 对于一些不经常更新的接口数据，可以将其缓存起来，下次请求时直接从缓存中获取，减少请求次数。实现方式很多，比如使用localStorage、sessionStorage、cookie，甚至serverWorker等。
  • 数据缓存 对于一些需要复杂计算的数据，也可以将其缓存起来，避免重复计算。

# 分片

对于一个计算比较密集的任务，可以将任务分片，分片执行，从而避免阻塞主线程。大致思路是利用requestIdleCallback API获取当前帧的空闲时间，将任务分片，在空闲时间执行。

# worker

这也是用以处理计算密集型任务的一种方式，可以将计算任务放在worker中执行，从而避免阻塞主线程。不过使用场景有限（对于笔者来说），我仅在大文件分片时这么用过，当然效果也是非常显著的。

# 动画

对于动画，最佳使用css3 的 transform 、 animation 、 transition 等属性，因为浏览器会对这些属性进行优化，这些属性性能很高。 若需要js控制，首推requestAnimationFrame API，（不推荐setTimeout等方式,若延时与屏幕帧率不同会有卡顿现象）。这里直接推荐gsap，这个动画库，性能非常好，且使用简单。
